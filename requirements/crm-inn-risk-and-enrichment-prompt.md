# Промт: расширение CRM — ИНН, идентификация юрлица, риск-скоринг, поиск

Документ для передачи в Cursor / другую ИИ-сессию или команде разработки. Цель — спецификация фич для **idev-crm** (Django + React), без привязки к конкретным внешним API на первом этапе.

---

## Контекст продукта

Внутренняя CRM: клиенты, сделки, задачи, бэклог, календарь, отчёты. Нужно усилить **идентификацию организаций** (в т.ч. для РФ и кросс-бордер сценариев), **поиск** и **оценку риска** лида/клиента, не ломая существующие модели и права доступа.

---

## Цели (MVP)

1. **Идентификаторы юрлица**
   - Поля на модели клиента (минимум):
     - `tax_id` (ИНН для РФ) — строка, нормализация: только цифры, длина 10 (ЮЛ) или 12 (ИП); валидация на бэкенде.
     - Опционально позже: `kpp`, `ogrn` / `ogrnip`, `legal_name` (как в реестре), `country` (ISO 3166-1 alpha-2).
   - **Уникальность**: при наличии непустого `tax_id` + `country` (или дефолт `RU`) — уникальная пара в БД; при конфликте — понятная ошибка при создании/импорте и подсказка «возможный дубликат».

2. **Поиск**
   - В списке клиентов и в глобальном поиске (если есть): поиск по **названию**, **ИНН** (частичное и полное совпадение), **email контактов** (как сейчас или расширить).
   - Backend: `SearchFilter` / `icontains` по нормализованному ИНН или отдельное поле `tax_id_normalized` (индекс) для быстрого точного поиска.

3. **Риск-скоринг (rule-based, без ML в MVP)**
   - Поля на клиенте:
     - `risk_score` — целое 0–100 или enum `low | medium | high | critical`.
     - `risk_notes` — краткое пояснение (текст).
     - `risk_overridden` — bool + `risk_override_by`, `risk_override_at` (если менеджер вручную зафиксировал уровень).
   - **Пересчёт по правилам** (примеры правил, настраиваемые константами или простой таблицей правил в коде):
     - Пустой ИНН при статусе не `lead` → +к риску.
     - Дубли по ИНН в системе → флаг + повышение риска.
     - Клиент без контактов с ролью ЛПР при активной сделке → средний риск.
     - (Опционально позже) вебхуки из внешних источников.
   - **Ручной оверрайд**: менеджер выставляет уровень/скор; при следующем пересчёте правила не затирают оверрайд, пока не снят флаг «снять оверрайд и пересчитать».

4. **UI**
   - Карточка клиента: блок «Реквизиты» — ИНН (маскированный показ опционально по роли), копирование в буфер.
   - Бейдж риска в списке клиентов и в шапке карточки (цвета согласованы с дизайн-токенами `--danger`, `--warning`, и т.д.).
   - Фильтр в списке клиентов: по диапазону `risk_score` или по уровню.
   - При создании клиента: live-подсказка «найден похожий клиент с таким ИНН» (debounce запрос к API).

5. **Аудит (минимум)**
   - Логирование смены `risk_score` / оверрайда (можно через `django-simple-history` или простую модель `ClientRiskAudit`).

---

## Не входит в MVP (явно отложить)

- Автоматическое подтягивание данных из **ФНС / ЕГРЮЛ** без юридической проработки и договоров.
- ML-модель скоринга.
- Массовый скоринг по всей базе по cron без явной команды админа (можно добавить management command позже).

---

## Технические требования

- **Backend**: Django REST, миграции, сериализаторы с валидацией ИНН; права: поле ИНН и финансовые поля — только ролям с флагом (если в проекте уже есть `Role`, расширить `can_view_sensitive_client_data` или аналог).
- **Frontend**: TypeScript, формы с валидацией, отображение ошибок API; не ломать существующий `Client` тип — расширить.
- **Производительность**: индекс на `(country, tax_id_normalized)` или уникальный partial index где `tax_id != ''`.
- **Локализация**: подписи полей RU/EN в `i18n` при необходимости.

---

## Критерии приёмки (чеклист)

- [ ] Создание клиента с валидным ИНН 10/12 цифр; отклонение невалидного ИНН с понятным сообщением.
- [ ] Невозможность создать второго клиента с тем же ИНН (при выбранных правилах уникальности).
- [ ] Поиск по ИНН из списка клиентов находит нужную запись.
- [ ] Риск отображается в списке и в карточке; ручной оверрайд сохраняется и не перезаписывается автопересчётом без явного действия.
- [ ] После изменения правил пересчёта (или вызова endpoint/command) скор обновляется для клиентов без оверрайда.
- [ ] Нет утечки ИНН в публичных API для ролей без доступа (проверить с тестовым пользователем).

```
Реализуй в репозитории idev-crm MVP по спецификации из файла requirements/crm-inn-risk-and-enrichment-prompt.md:

Не подключай внешние реестры ФНС в этом PR. Следуй стилю существующего кода; не делай лишних рефакторингов вне области задачи.
```

## Рынок и лучшие практики (обзор по открытым источникам)

Ниже — консолидированные ожидания к «зрелой» B2B CRM и enterprise-подходам, которые стоит учитывать при расширении **idev-crm**, даже если часть функций откладывается. Источники для ориентира (не как юридическая норма): обзоры must-have фич, сравнения крупных платформ, документация по predictive scoring.

### Что повторяют обзоры B2B CRM (2024–2026)

- **Воронка и видимость pipeline** — явные стадии, конверсия между стадиями, «застрявшие» сделки; weighted forecast по вероятности и сумме ([b2bdaily.com](https://b2bdaily.com/martech/the-20-essential-features-of-a-high-performing-crm/), [erfolk.com](https://erfolk.com/blog/pipeline-management-best-practices/)).
- **Автоматизация и скорость реакции** — напоминания, триггеры по смене стадии, «малые смысловые workflow» с понятной целью, а не «комбайн из 20 шагов» ([prismatic-technologies.com](https://prismatic-technologies.com/blog/10-must-have-crm-features-to-boost-sales-in-2025/)).
- **Качество данных и обогащение** — валидация полей, дедупликация, единая «золотая запись» клиента ([prismatic-technologies.com](https://prismatic-technologies.com/blog/10-must-have-crm-features-to-boost-sales-in-2025/)).
- **Аналитика и атрибуция** — дашборды, цикл сделки, время ответа, привязка к UTM/кампаниям где уместно ([isitdev.com](https://isitdev.com/crm-features-top-2025/)).
- **Мобильность** — быстрый доступ к карточке и лог звонков с поля (часто упоминается для полевых продаж) ([prismatic-technologies.com](https://prismatic-technologies.com/blog/10-must-have-crm-features-to-boost-sales-in-2025/)).
- **Омниканальность** — единая история по email / чату / тикетам (для внутренней CRM — хотя бы **единая лента активностей** по клиенту и сделке).

### Что делают enterprise-игроки (Salesforce, Microsoft Dynamics, Oracle — как концепции)

- **Lead / opportunity scoring** — числовой балл, **объяснимые факторы** (какие поля и сигналы подняли/опустили скор), тренд «улучшается / падает», частота обновления ([Microsoft Learn — lead scoring](https://learn.microsoft.com/en-us/dynamics365/sales/digital-selling-scoring)).
- **Предиктивные модели** в enterprise обычно требуют **объём обучающих данных** и лицензионных ограничений — для внутренней CRM разумнее начать с **rule-based + аудит**, а ML вынести в отдельную фазу ([Microsoft Learn — predictive](https://learn.microsoft.com/en-us/dynamics365/sales/work-predictive-lead-scoring)).
- **Customer 360 / health** — склейка сигналов из сделок, задач, оплат (если появятся), поддержки; **health score** для существующих клиентов (отдельно от риска лида) ([influencers-time.com](https://www.influencers-time.com/choosing-predictive-crm-extensions-key-evaluation-criteria-for-2026-success/)).

### Модули «как у HubSpot / Salesforce» (ориентир по границам продукта)

Крупные экосистемы делят пространство на **Sales**, **Service**, **Marketing**, **Ops/автоматизация**, **отчёты**, **кастомные объекты**. Для **idev-crm** разумный ориентир без раздувания:

| Область | Практика рынка | Реалистично для idev-crm |
|--------|----------------|---------------------------|
| Sales Cloud | Pipeline, forecast, CPQ | Уже есть сделки/канбан — усилить **обязательные поля по стадиям**, **weighted pipeline** в отчётах |
| Service | Тикеты, SLA | Позже: лёгкие «кейсы» или связь с внешним helpdesk |
| Ops | Workflows, дедуп | **Дедуп по ИНН**, webhooks, audit trail |
| Data | Единый профиль клиента | **Лента активностей**, связь контакт ↔ сделка ↔ задачи |
| AI | Скоринг, подсказки | Только после **объяснимого** rule-based слоя и норм данных |

Сравнительные обзоры модулей: [crm.org — HubSpot vs Salesforce](https://crm.org/crmland/hubspot-vs-salesforce), [themoneyplaybooks.com](https://themoneyplaybooks.com/comparison/salesforce-vs-hubspot-2026/).

## Дополнения к продуктовой спецификации (Phase 2+)

Имеет смысл **дописать в бэклог** отдельными эпиками (не обязательно в первом MVP ИНН/риск).

### 1. Золотая запись и дедупликация

- **Match keys**: ИНН + страна; дополнительно — нормализованный домен email, телефон E.164.
- **Merge flow**: два дубликата → мастер-запись, слияние контактов и ссылок на сделки с **аудитом** (кто слил, когда).
- **Soft-delete / «помечен дубликатом»** вместо жёсткого удаления.

### 2. Иерархия и связи «материнская ↔ дочерние» организации

- Для холдингов: `parent_client_id`, роллап суммы pipeline (опционально), единый риск на группу.

### 3. Лента активностей (Customer timeline)

- Единый хронологический вид: создание клиента, смена стадии сделки, задачи, комментарии бэклога, встречи из календаря.
- Фильтры по типу события и автору — снижает «слепые зоны» между модулями.

### 4. Разделение **Lead risk** vs **Customer health**

- **Risk** (лид / ранняя стадия): правила, ИНН, полнота данных, дубли.
- **Health** (после закрытия в пользу): NPS, просрочки оплат, объём открытых задач support, «давно не было контакта» — отдельный score и дашборд «кого спасать».

### 5. Объяснимый скоринг (Explainability)

- Для каждого пересчёта хранить **JSON или строку**: `factors: [{ code, weight, detail }]`, показывать в UI «почему такой риск» (как эталон у enterprise scoring).
- Ручной оверрайд с **комментарием обязателен** при уровне critical.

### 6. Соответствие данным (РФ / GDPR mindset)

- **Цель обработки** и срок хранения для ИНН/регистрационных данных (даже во внутренней CRM).
- Маскирование ИНН в UI и экспортах для ролей без права; в CSV отчётах — отдельная политика (как уже задумано для чувствительных полей).
- Логи доступа к «чувствительной карточке» (опционально Phase 3).

### 7. Интеграции и обогащение (после юридики)

- Подключение **официальных** источников ЕГРЮЛ/ФНС только через договор и явный **флаг согласия** в карточке.
- Webhook входящий: обновление статуса юрлица из внешней системы → пересчёт риска.

### 8. Прогноз и управление pipeline (уже близко к текущему продукту)

- Weighted amount по сделкам: `value * probability / 100`.
- Воронка: **SLA по стадии** (дней в стадии > N → задача руководителю).
- Snapshots pipeline на дату (для отчёта «как выглядел квартал») — отдельная таблица или nightly job.

### 9. Мобильный сценарий (минимум)

- Адаптивный канбан уже есть; для поля: **быстрое добавление заметки / задачи** с телефона без полной формы клиента.

### 10. Качество внедрения автоматизаций (best practice)

- Один workflow — одна измеримая цель; поле «причина срабатывания» в логе; возможность **откатить** последнее автодействие для админа.
